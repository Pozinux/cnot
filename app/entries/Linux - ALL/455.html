Le droit de lecture :
  
Pour un fichier = consultable via l'application qui convient
Pour un répertoire = on peut lister son contenu mais seulement les noms de fichiers (ls)
  
Le droit d'écriture :
  
Pour un fichier = possibilité de modifier le fichier
Pour un répertoire = possibilité d'y créer ou d'y supprimer un fichier ou répertoire
  
Le droit d'exécution :
  
Pour un fichier = autorise l'utilisation de fichiers exécutable (les données à l'intérieur du fichier (des commandes par exemple) pourront être exécutées).
Pour un répertoire = l'utilisateur est autorisé à se positionner dedans (avec la commande cd par exemple) et lister les fichiers, leur permissions etc (ls -l)
  
Si le owner d'un fichier a seulement le droit read et pas write sur un fichier, il peut quand même le modifier avec VI mais VI va faire une sécurité en demandant si on veut vraiment le modifier et sortir en utilisant "!".
Le owner d'un fichier a le droit de tout faire même si les droits sont à 0 par exemple avec le fichier shadow qui n'a aucun droit et pourtant root peut le modifier :
  
[root@centos3 fd]# ls -l /etc/shadow
----------. 1 root root 1248 23 janv. 13:58 /etc/shadow
  
ATTENTION : Si un user a les droits en écriture sur un répertoire, il pourra supprimer un fichier dedans même s'il ne lui appartient pas ! (solution : voir Sticky bit plus bas)
  
Attention : Si un répertoire à le droit X par exemple pour other mais qu'en faisant cd dedans on a un "permission denied" c'est sûrement car un répertoire au dessus n'a pas le droit X pour others. Remonter l'arborescence avec ll -d.
  
A SAVOIR PAR COEUR :
  
  
u=owner (user)
g=group
o=other
  
Ajouter ou supprimer un droit (par exemple le droit d'exécution au group). Va ajouter/supprimer sans changer ce qui existe déjà :
  
# chmod g+x fic1    // ajout exécution pour le group
# chmod ug-w fic1   // supprimer l'écriture pour le owner et le group
# chmod +r fic1     // ajout lecture pour tous
  
Affecter un droit (s'il y avait par exemple ci-dessous le droit de lecture pour les autres (other) et rien pour le group, il est replacé par l'écriture pour les autres et attribué pour le group. Pour le owner ça ne bouge pas car on a pas précisé u) :
  
# chmod go=w fic1
  
On peut aussi affecter en octal directement pour les trois :
  
# chmod 644 fic1
  
  
DROITS SPECIAUX (setuid, setgid, sticky bit)
  
Note importante : ces droits sont représentés par "s" (setui et setgid) et "t" (sticky bit) et remplacent le droit d'exécution mais peuvent être en majuscule pour désigner que le x qu'il remplace n'est pas présent. Donc pour un setuid ou setgid, s'il y a un S c'est qu'il y a un problème car on ne devrait positionner ces bits que sur des fichiers exécutables.
  
Note importante : On ne peut pas clearer un setuid ou setgid bit par la méthode numérique. Il faut utiliser chmod u-s (retirer le setuid) et chmod g-s (retirer le setgid) pour ça.
C'est ce qu'il y a de marqué dans le man de chmod mais j'ai l'impression de pouvoir sur ma Centos.
Par contre, pour le sticky bit, il suffit de faire chmod 0755.
  
SETUID (sur les fichiers)
  
A la place du x d'exécution du owner on a un "s". C'est le setuid bit :
  
[tim@centos3 ~]# ls -l /usr/bin/passwd
-rwsr-xr-x. 1 root root 27832 Jun 10  2014 /usr/bin/passwd
  
On appelle ça le droit 4000. Pour le positionner sur fic1 par exemple + les droits en 755 on fait 4755 :
  
[tim@centos3 ~]$ ls -l
total 0
-rw-rw-r-- 1 tim tim 0 Sep 26 11:43 fic1
  
[tim@centos3 ~]$ chmod 4755 fic1
  
[tim@centos3 ~]$ ls -l
total 0
-rwsr-xr-x 1 tim tim 0 Sep 26 11:43 fic1
  
  
Cas concret : la commande "passwd" modifie un fichier (etc/shadow) qui est propriétée de root mais tout le monde peut le modifier (pour modifier son propre mdp) avec la commande "passwd".
Signifie que lorsque qqn exécute cette commande, il devient le owner (donc là il devient root) le temps de l'exécution. Par exemple pour que le mdp puisse être écrit dans le /etc/shadow quand un utilisateur utilise la commande passwd pour changer son mdp. On parle de uid réel (ici tim) et uid effectif (ici root) pendant la commande.
  
[root@centos3 ~]# whereis passwd
passwd: /usr/bin/passwd /etc/passwd /usr/share/man/man1/passwd.1.gz /usr/share/man/man5/passwd.5.gz
  
[root@centos3 ~]# ll /usr/bin/passwd
-rwsr-xr-x. 1 root root 27832 10 juin   2014 /usr/bin/passwd
  
alors que le fichier que la commande modifie :
  
[root@centos3 ~]# ll /etc/shadow
----------. 1 root root 1248 23 janv. 13:58 /etc/shadow
 
  
SETGID (sur les fichiers + répertoires)
  
A la place du x d'exécution du group on a un "s". C'est le setgid bit.
  
Si on le positionne sur un répertoire, les fichiers qui seront créés dans ce répertoire auront le même group d'appartenance que celui du répertoire même si le fichier est créé avec un autre user.
  
On appelle ça le droit 2000. Pour le positionner sur rep1 par exemple + les droits en 755 on fait 2755 :
  
[tim@centos3 ~]$ mkdir rep1
  
[tim@centos3 ~]$ ls -l
total 0
drwxrwxr-x 2 tim tim 6 Sep 26 11:52 rep1
  
[tim@centos3 ~]$ chmod 2775 rep1
  
[tim@centos3 ~]$ ls -l
total 0
drwxrwsr-x 2 tim tim 6 Sep 26 11:52 rep1
  
[tim@centos3 ~]$ exit
logout
  
[root@centos3 ~]# touch /home/tim/rep1/fic1
  
[root@centos3 ~]# ls -l /home/tim/rep1/fic1
-rw-r--r-- 1 root tim 0 Sep 26 11:53 /home/tim/rep1/fic1
  
  
Cas concret : Un répertoire partagé dans lequel plusieurs personnes peuvent créer des fichiers et utiliser ceux des autres.
  
STICKY BIT (sur les répertoires)
  
Le droit d'écriture sur un répertoire signifie que tout le monde peut créer et supprimer les fichiers de ce répertoire. Cela peut être dangereux car qqn peut supprimer un fichier d'un répertoire alors que le fichier ne lui appartient pas.
  
Lorsque ce droit est positionné sur un répertoire, il interdit la suppression d'un fichier qu'il contient à tout utilisateur autre que le propriétaire du fichier ou root. Néanmoins, il est toujours possible pour un utilisateur possédant les droits d'écriture sur ce fichier de le modifier (par exemple de le transformer en un fichier vide). La création de nouveaux fichiers est toujours possible pour tous les utilisateurs possédant le droit d'écriture sur ce répertoire.
  
Note historique : Si l'on positionne le sticky bit sur un fichier, l'utilisation est tout autre. Le bit collant (sticky bit) indique alors que ce fichier doit encore rester en mémoire vive après son exécution. Le but était, à l'époque, d'améliorer les performances en évitant de charger/décharger un fichier de la mémoire (par exemple, un exécutable ou une bibliothèque logicielle). Le terme sticky (collant) voulait dire que le fichier restait collé en mémoire.
  
Son flag est le t ou T, qui vient remplacer le droit d'exécution x des autres utilisateurs que le propriétaire et ceux appartenant au groupe du fichier, de la même façon que les droits SUID et SGID. La capitale fonctionne aussi de la même façon, elle est présente si le droit d'exécution x caché n'est pas présent.
  
Un fichier avec les droits
  
-rwxr-xr-x
  
auquel on ajoute le droit sticky bit aura donc la notation
  
-rwxr-xr-t
  
Le droit sticky bit possède la valeur octale 1000.
Exemple : - r w x r - x r - t correspond à 1755.
  
Pour le positionner donc 
  
# chmod 1755 rep1
  
Cas concret :
  
Le sticky bit est souvent utilisé pour le répertoire /tmp, qui doit être accessible en écriture par tous les utilisateurs, sans que ceux-ci ne puissent supprimer les fichiers des autres :
  
drwxrwxrwt 86 root root 20480 tmp

ACL : Pour afficher les droits (et les ACL d'un fichier) :
  
Si aucun ACL est positionné, on a les droits habituels qui s'affichent. Sinon, la ligne group n'indique pas les droits mais le mask ACL. Voir note ACL : ACL - Listes de contrôle d'accès
  
[root@centos2 ~]# getfacl toto.php
# file: toto.php
# owner: root
# group: root
user::rw-
group::r--
other::r--
  
Si sur un répertoire on a les sticky, setuid et setgid bits positionnés, on aura un champ flag qui apparaitra :
  
[root@centos2 ~]# chmod u+s rep1/   ---> positionner le setuid
[root@centos2 ~]# chmod g+s rep1/   ---> positionner le setgid
[root@centos2 ~]# chmod 1777 rep1/  ---> positionner le sticky bit
  
[root@centos2 ~]# getfacl rep1/
# file: rep1/
# owner: root
# group: root
# flags: sst     ---> setuid/setgid/sticky-bit
user::rwx
user:tim:rwx
group::r-x
mask::rwx
other::rwx

Note : Le find fait pareil mais en donnant plus d'infos
  
# find /appl -ls | more
  
et plus précis pour voir les group par exemple :
  
[root@~]# find /appl/ -ls | awk '{print $6}'| sort | uniq |more
httpd

root
