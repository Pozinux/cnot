Sous CentOS c'est le service rsyslog (attention j'ai vu syslog sous Redhat 5). Sous d'autres distributions ça peut-être syslog ou syslogng.
Le journal de bord prinicipal pour Redhat/CentOS est /var/log/messages.
Pour Debian/Ubuntu /var/log/syslog.
  
Rsyslog est un outils d'enregistrement et de gestion des journaux d'événements qu'il enregistre dans /var/log par défaut.
Les services et applications écrivent généralement dans syslog mais certains ont leur propre system de logs comme par exemple apache.
  
Rsyslog implémente le protocol syslog fonctionne en mode client server.
Les journaux peuvent être centralisés sur un serveur syslog.
  
Les sous-systèmes :
  
  
auxquels on donne une facility qui défini un niveau de priorité du sous-system :
  
  
Si par exemple on met kern.info, on aura tous les messages de type kern.info + kern.debug. C'est décroissant.
  
Tout ça se configure dans /etc/rsyslog.conf.
  
Exemple de régle :
  
# Log all kernel messages to the console.
# Logging much else clutters up the screen.
#kern.*                                                 /dev/console         // tous les messages du kernel
  
# Log anything (except mail) of level info or higher.
# Don't log private authentication messages!
*.info;mail.none;authpriv.none;cron.none                /var/log/messages     // tous les messages de facility info sauf ceux de mail, authpriv et cron
  
# Log all the mail messages in one place.
mail.*                                                  -/var/log/maillog     // sans synchronisation ??
  
  
Comme les événements de démarrage n'ont pas de sous-system particuliers, on va utiliser local7 :
  
# Save boot messages also to boot.log
local7.*                                                /var/log/boot.log
  
Exemple concret, on souhaite avoir les logs de SSH dans un log particulier syslog (par défaut ssh est géré par syslog avec le sous-system AUTHPRIV) :
  
[root@rhel1 ~]# grep -A3 -B3 log /etc/ssh/sshd_config
...
#RekeyLimit default none
  
# Logging
#SyslogFacility AUTH
SyslogFacility AUTHPRIV
#LogLevel INFO
  
# Authentication:
...
  
On modifie, on dit à syslog d'écouter ssh via le local0 puis on redémarre le service sshd :
  
[root@rhel1 ~]# grep -A3 -B3 log /etc/ssh/sshd_config
...
#RekeyLimit default none
  
# Logging
#SyslogFacility AUTH
#SyslogFacility AUTHPRIV
SyslogFacility local0
#LogLevel INFO
  
# Authentication:
...
  
A partir de là, les connexions ssh ne sont plus loguées dans /var/log/secure. 
  
On configure local0 dans rsyslog puis redémarrer le service rsyslog :
  
[root@rhel1~]# vi /etc/rsyslog.conf
  
# Save sshd sessions
local0.*                                                /var/log/sshd.log

AUTRE EXEMPLE CONCRET :
  
Je souhaite envoyer les messages de criticité "info" de ma Centos vers ma Redhat (qui fera donc office de serveur syslog).
  
Sur la Centos :
  
DANS LA SECTION RULES, je rajoute cette ligne :
 
@@ pour une ip
@ pour un nom dns
le port par défaut est le 514 donc pas obligé de le mettre
  
[root@centos2 rsyslog.d]# vi /etc/rsyslog.conf
  
# Send to other server
*.info                                                   @@192.168.0.16:514
  
Restart rsyslog service.
  
Sur la Redhat :
  
- Désactiver le firewalld
- Modifier rsyslog pour qu'il écoute les requête syslog sur le port par défaut en TCP, décommenter ces deux lignes :
  
[root@rhel1 ~]# vi /etc/rsyslog.conf
  
...
# Provides TCP syslog reception
$ModLoad imtcp
$InputTCPServerRun 514
...
