Attention, cela fonctionne que pour des utilisateurs en connexion ssh à distance pas pour du local.
  
  
L'avantage du sudo même si on donne sudo root (la commande "/bin/su -") est que le user est logué (apparait dans les logs) comme étant passé root et que ses opérations sont loguées aussi).
  
La commande su est journalisée dans /var/log/secure mais on ne voit que qui est passé root (ou un autre user) et quand il l'a quitté.
On ne voit pas les commandes qui ont été passées lorsque la personne étaient en su root.
Par contre, si on utilise sudo et non pas su, là tout est journalisé même les commandes passées en tant que sudo
  
  
Pour ne pas avoir à écrire sudo devant chaque commande, on peut lancer un shell root de plusieurs façons (sans parler des façons avec su )  :
  
$ sudo -s     -------> lance un shell root mais avec l'environnement du user actuel
  
ou beaucoup mieux :
  
$ sudo -i     --------> lance un shell root avec l'environnement root   
  
Note : On peut faire pareil pour un utilisateur précis :
  
$ sudo -i -u tim
  
  
Lister les commandes que peut exécuter l'utilisateur connecté :
  
[tim@centos ~]$ sudo -l
Matching Defaults entries for tim on this host:
    requiretty, !visiblepw, env_reset, env_keep="COLORS DISPLAY HOSTNAME HISTSIZE INPUTRC KDEDIR LS_COLORS MAIL PS1 PS2 QTDIR USERNAME LANG LC_ADDRESS LC_CTYPE LC_COLLATE LC_IDENTIFICATION LC_MEASUREMENT LC_MESSAGES LC_MONETARY LC_NAME LC_NUMERIC LC_PAPER LC_TELEPHONE
    LC_TIME LC_ALL LANGUAGE LINGUAS _XKB_CHARSET XAUTHORITY"
  
User tim may run the following commands on this host:
    (ALL) /usr/sbin/userdel
  
En root, lister les commandes que peut exécuter l'utilisateur tim :
  
sudo -l -U tim
  
Pour donner des droits sudo, éditer le fichier /etc/sudoers avec la commande visudo en tant que root.
Ne pas éditer le fichier directement avec vi (voir pourquoi dans le man de visudo).
  
Syntaxe générale du fichier sudoers :
  
usernames/group servername = (usernames command can be run as) command1,command2
  
Permettre à l'utilisateur admin d'exécuter la commande /user/sbin/adduser en tant que root sur la machine centos (besoin du mot de passe) :
  
admin   centos=(root)   /usr/sbin/adduser
  
Afin d'exécuter la commande sous le compte admin :
  
sudo /usr/sbin/adduser
  
Permettre à l'utilisateur tim d'exécuter la commande /user/sbin/adduser en tant que n'importe qui sur n'importe quelle machine (pas besoin du mot de passe) :
  
tim   ALL=(ALL)   /usr/sbin/adduser
  
Sudo root : Permettre à tim de passer root en faisant "su -" :
  
tim   hostname_serveur=/bin/su -
  
Voilà ce qui se produit quand on n'a aucun droit sudo :
  
[tom@centos admin]$ sudo -l
[sudo] password for tom:
Sorry, user tom may not run sudo on centos.
  
OU BIEN si on essaye une commande en sudo :
  
[tom@centos ~]$ sudo ifconfig
[sudo] password for tom:
tom is not in the sudoers file.  This incident will be reported.
  
Note : le user root aura un mail (que l'on voit avec la cde mail) reportant l'incident.
  
Si je n'ai pas les droits sur une certaine commande mais que je suis dans le sudo :
  
[admin@centos tom]$ sudo /usr/sbin/userdel
[sudo] password for admin:
Sorry, user admin is not allowed to execute '/usr/sbin/userdel' as root on centos.
  
Contrôler le fichier de configuration et repérer les erreurs de syntaxe :
[root@centos ~]# visudo -c
/etc/sudoers: parsed OK
  
Pour toutes les commandes que va taper toto (avec sudo devant ou après s'être authentifié avec sudo) cumuleront ses droits à lui et ceux de jerome (car il les exécute en tant que Jerome). Plus concrètement il pourra tout faire dans le /home de Jerome par exemple :
  
toto ALL=(jerome) ALL
  
Exemple de configuration avec des alias
# Host alias specification
  
# User alias specification
User_Alias STAFF=gaelle,barnabe,chris
  
# Cmnd alias specification
Cmnd_Alias NET=/bin/ping,/usr/bin/traceroute,/usr/bin/ftp,/usr/bin/nmap
  
# User privilege specification
root ALL=(ALL) ALL
STAFF ALL=(ALL) NET
  
Grâce à ce fichier sudoers, les utilisateurs gaelle,barnabe et chris peuvent désormais exécuter des pings, des traceroutes, des transferts FTP et des nmap comme si ils étaient root.

Vous pouvez trouver de nombreux exemples de la syntaxe de sudoers dans la page de manuel (man sudoers) avec les explications correspondantes, et vous verrez que de nombreuses possibilités existent en plus de celles que nous venons de voir.

Utilisation
Vous êtes maintenant à même de définir votre politique sudo sur la machine, toutefois pour rester dans le concret, la première utilisation qu'on en fait, c'est d'autoriser son utilisateur à effectuer des opérations sans passer root. Pour cela il suffit d'ajouter la ligne suivante avec visudo :
  
Cedric ALL=(ALL) NOPASSWD:ALL
  
Désormais, pour passer root, il suffira à Cédric de taper la commande sudo suivie de la touche "ENTREE". On ne lui réclamera pas de mot de passe à cause de la directive NOPASSWD. Sans cette directive son mot de passe lui serait réclamé (j'insiste: le sien, pas celui du compte root).
Mais il y a une autre manière bien commode d'utiliser la commande: lorsqu'on souhaite effectuer une commande ponctuelle en tant que root, il suffit de faire précéder la commande en question par "sudo". Voici ce que ca donne :
  
$ sudo rm mp3blaster-3.1.3.tar.gz
 
Cette commande efface le fichier même s'il est interdit à l'utilisateur de le faire et sans demander de mot de passe ou de confirmation. C'est pourquoi il faut être très prudent quand on utilise sudo, surtout avec la directive :
  
"NOPASSWD:ALL"
 
Vous pouvez exécuter d'autres commandes privilégiées en utilisant sudo dans un délai de cinq minutes sans être ré-invité à entrer un mot de passe. Toutes les commandes s'exécutent en tant que sudo sont enregistrées dans le fichier journal /var/log/messages.
  
Il est possible aussi de lancer un shell administrateur pour éviter d'entrer sudo devant chaque commande en utilisant la commande :
  
$ sudo -s     -------> lance un shell root mais avec l'environnement du user actuel
  
ou
  
$ sudo -i     --------> lance un shell root avec l'environnement root   
  
Il est possible de faire du sudo pour une ligne de commande avec paramètres :
  
Dans le fichier sudo :
  
admin   centos=(root)   /bin/rm /home/root/toto2018.txt
  
et la commande résultante :
  
# sudo rm /home/toto2018.txt
  
Attention, si l'utilisateur entre une commande pas exactement pareil même si c'est la même chose, elle ne sera pas acceptée :
  
cd /home/root/
sudo rm toto2018.txt
  
Il est aussi possible de faire des wildcards :
  
* : n'importe quel caractère
  
admin   centos=(root)   /bin/rm /home/root/toto*.txt
  
  
Some of the Situations, and their corresponding ‘sudo‘ line:
  
Q1. You have a user mark which is a Database Administrator. You are supposed to provide him all the access on Database Server (beta.database_server.com) only, and not on any host.
  
mark beta.database_server.com=(ALL) ALL
  
Q2. You have a user ‘tom‘ which is supposed to execute system command as user other than root on the same Database Server, above Explained.
  
mark beta.database_server.com=(tom) ALL
  
Q3. You have a sudo user ‘cat‘ which is supposed to run command ‘dog‘ only.
  
mark beta.database_server.com=(cat) dog
  
Q4. What if the user needs to be granted several commands?
  
If the number of commands, user is supposed to run is under 10, we can place all the commands alongside, with white space in between them, as shown below:
  
mark beta.database_server.com=(cat) /usr/bin/command1 /usr/sbin/command2 /usr/sbin/command3 ...
  
If this list of command varies to the range, where it is literally not possible to type each command manually we need to use aliases. Aliases! Yeah the Linux utility where a long-lengthy command or a list of command can be referred as a small and easy keyword.
  
A few alias Examples, which can be used in place of entry in ‘sudo‘ configuration file.
  
User_Alias ADMINS=tom,jerry,adam
user_Alias WEBMASTER=henry,mark
  
WEBMASTERS WEBSERVERS=(www) APACHE
Cmnd_Alias PROC=/bin/kill,/bin/killall, /usr/bin/top
  
It is possible to specify a System Groups, in place of users, that belongs to that group just suffixing ‘%’ as below:
  
%apacheadmin WEBSERVERS=(www) APACHE
  
Q5. How about executing a ‘sudo‘ command without entering password?
  
We can execute a ‘sudo‘ command without entering password by using ‘NOPASSWD‘ flag.
  
adam ALL=(ALL) NOPASSWD: PROCS
  
Here the user ‘adam‘ can execute all the commands aliased under “PROCS”, without entering password.
  
"sudo” provides you a robust and safe environment with loads of flexibility as compared to ‘su‘. Moreover “sudo” configuration is easy. Some Linux distributions have “sudo” enabled by default while most of the distros of today needs you to enable it as a Security Measure.
